Welcome to Yaflengine site.

Here you'll find information about the project, download links to the latest releases and links to API documentation.

Yaflengine stands for "Yet Another Fuzzy Logic Engine". It is entirely written in C++ and compiles under Linux and Windows (via MinGW). It provides a series of classes to help creating fuzzy logic models and perform calculations. Fuzzy models are stored in text files using the .ini file format, but can also be created at runtime. Value functions (truth functions if you wish) can be defined, although common ones are already built in the library (Gaussian bell, s-shaped, triangular, ...).

Rules syntax is very similar to plain english, so it is very easy to understand and learn. Take a look at the followin example: 

if mode.takeoff then throttle.full
if mode.takeoff and speed.not.stall then pitch.little.up
if mode.takeoff and airborne.true then gear.retract
if mode.takeoff and airborne.true then mode.flying

The example shows a possible model for an UAV take-off procedure. You may have noticed two things about the syntax:

1. As you may have noticed, the syntax understands several modifiers such as .not, .very or .little.
2. Sets can act as inputs, outputs or both. this, I think (I may be wrong here) this is not a common feature of fuzzy engines.

Library history

I wrote this library around 2005 or so for the solely purpose of understanding fuzzy logic. It has ever since gone through several updates.

The first version lacked the rule modifiers and Set dual roles in a single model. To use a Set both as an input and output one had to create two independent models, load both independently, and run them in the required order to achieve what was needed. It was very messy. On top of that, it was written for the C++Builder compiler and it included features only understood by it (__property, for example).

The first update changed the code to make it compile by the GNU C++ compiler g++ both in Linux and Windows. The library became multiplatform (or dual-platform at least).

The next update introduced modifiers in the rule syntax.

The major change to this date came in adding the posibility of having sets play an input and output role within the same model. This meant changing the calculation chain rules. The classes TExecData and TFuzzyExec were born. The engine is now inteligent enough to know which sets are pure inputs, pure outputs and input/output and generate the correct calculation chain to generate the final outputs. In fact one can concatenate any number of calculation levels on the chain, and it is completely transparent to the user.
